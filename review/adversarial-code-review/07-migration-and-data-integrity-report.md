For both upgraded copies:

- `PRAGMA foreign_key_check` returned no rows.

These fixtures were deliberately small and synthetic. They are positive evidence, not a substitute for the permanent matrix in SA-ACR-013.

## Transaction and durability review

Positive controls:

- GRDB migrations are applied through `DatabaseMigrator` and foreign keys are enabled per connection.
- Backup service has unusually good durability/failure tests: temporary cleanup, snapshot/blob consistency, atomic manifest publication, and retention ordering.
- Unknown migration history is refused without auto-erasing real data.
- Repository cascades and FTS replacement have focused tests.

Material gaps:

- `PreMigrationSnapshot.captureIfUpgrading` is `try?`; migration proceeds when snapshot creation fails, including possible no-space conditions.
- Shipping-version fixtures are not permanent CI assets (SA-ACR-013).
- Document import's filesystem copy and database/extraction state are not one durable transaction (SA-ACR-006).
- Document and drafting exports are non-atomic (SA-ACR-009).
- Audit-event writes in many critical paths are best-effort, so an action may complete without a matching audit record.

## Data-loss and corruption risks

1. Managed blob/index mismatch or corrupt-path reuse (SA-ACR-006).
2. Loss of an existing export during replacement failure (SA-ACR-009).
3. Unsupported historical data shape not represented by fresh/toy migration tests (SA-ACR-013).
4. In-memory fallback after all on-disk stores fail is user-visible in the current shell, but real recovery and preventing durable work in the fallback store require signed-app fault testing.

## Recommended migration gates

- Generate fixtures using every supported release's actual store code: at minimum v1.4.1, v1.5.2, v1.8, v2.0.0, v2.1.0, v2.1.3, and latest-1.
- Seed all optional/null/duplicate/soft-deleted states, FTS/chunks/embeddings, source sets/versions, attachments, billing, and network/audit rows.
- Assert row-level business invariants, not only table existence.
- Run integrity, foreign-key, FTS, and full orphan queries before and after migration and after a second idempotent open.
- Fault migration midway and prove original database/snapshot recovery.
- Decide and document whether snapshot failure blocks migration for privileged user data.
